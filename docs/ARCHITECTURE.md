# Architecture

## Overview

Covantic is a parametric insurance protocol for AI agents on Solana. The system consists of five main components:

### 1. Solana Program (Anchor)

On-chain smart contract managing:

- Protocol configuration and vault state
- Oracle-signed **RiskAttestation** PDAs (one per agent)
- Insurance policy lifecycle (create, cancel, expire)
- Claim submission (`oracle_submit_claim`) + verified payout (`verify_and_payout`)
- USDC staking pool with per-stake reward accumulator

### 2. Backend API (Fastify)

Off-chain service providing:

- AI risk scoring engine (7-factor weighted model, Redis-cached)
- Claim verification oracle — per-trigger verifiers in `services/verifiers/` (exploit, oracle manipulation, agent error, governance attack)
- Transaction monitoring via Helius webhooks, with a signed internal alert bus (`services/alert-bus.ts`)
- Real-time WebSocket notifications (`claims:feed`, `vault:stats`, `monitoring:alerts`)
- Background workers:
  - `expiry-crank` — on-chain `expire_policy` for every stale policy
  - `solvency-checker` — on-chain vault read, publishes solvency alerts
  - `analytics-aggregator` — hourly vault snapshots
  - `policy-indexer` — mirrors on-chain `InsurancePolicy` accounts into Postgres
  - `claim-keeper` — drives the submit + payout pipeline

### 3. Frontend (Next.js)

Dashboard application with:

- Agent risk assessment interface
- Policy management (buy, view, cancel) — reads directly from chain
- Staking pool management (stake / request-unstake / execute-unstake / claim-rewards)
- Real-time claims feed with the verification pipeline bound to live WebSocket state
- `/fleet` — live panel over the autonomous agent fleet. Fleet agents exercise
  the claim pipeline end-to-end by generating real on-chain traffic —
  including deliberate failures (SPL Memo v2 with non-UTF-8 payload →
  `InstructionError::InvalidInstructionData`) so the AgentError verifier's
  `failed_tx` branch has material to score. Failure strategies live in
  `packages/api/src/services/fleet/failures.ts`.
- `/demo` — one-click trigger simulation for screen recording

### 4. SDK Plugin (separate repo)

Solana Agent Kit integration, shipped as [`@covantic/solana-sdk`](https://github.com/mihailShumilov/covantic-solana-sdk), enabling AI agents to:

- Check risk scores programmatically
- Purchase insurance before DeFi operations
- Submit claims automatically
- Query active policies

### 5. Shared Package

Cross-package types, constants, and utility functions ensuring type safety across the entire stack.

## Data Flow

```
Agent → /api/risk/:agent (7-factor score + tier)
          |
          v
Buyer  → /api/policies/quote
          |   (server publishes oracle-signed RiskAttestation PDA)
          v
Buyer  → createPolicy on-chain (program reads tier from attestation PDA)
          |
          v
Helius webhook → /api/monitoring/webhook → TransactionMonitor
          |           (anomaly → signed alert on monitoring:alerts)
          v
claim-keeper → oracle_submit_claim (on-chain)
          |           (lock period per trigger type)
          v
claim-keeper → verify_and_payout (on-chain)
          |           (USDC → holder ATA, policy.state = ClaimPaid)
          v
policy-indexer mirrors new state → /claims WebSocket feed → UI
```

## Attestation Handoff

`create_policy` no longer accepts a buyer-supplied risk tier. The flow:

1. Client calls `POST /api/policies/quote` with coverage + duration + agent.
2. The API checks the agent's latest risk assessment (must be non-EXTREME and ≤ 10 min old).
3. `AttestationPublisher` publishes or refreshes an on-chain `RiskAttestation` PDA signed by the oracle,
   with an `expires_at` inside `ATTESTATION_MAX_VALIDITY_SECONDS`.
4. The API returns the full `PremiumQuote` including `attestationPda` + `attestationExpiresAt`.
5. Client signs `createPolicy` including the `attestation` account. The on-chain program enforces
   `attestation.agent == agent_address && now <= expires_at` and sources the tier from the PDA.

Quote error codes (machine-readable): `ASSESSMENT_REQUIRED`, `AGENT_UNINSURABLE`, `ASSESSMENT_STALE`,
`ATTESTATION_PUBLISH_FAILED`.

## Claim Pipeline

Real (87–88 char Base58) trigger signatures route through `claim-oracle.ts` to per-trigger verifiers
in `packages/api/src/services/verifiers/`. Each verifier inspects the parsed transaction (pre/post
token balance diffs, fee payer analysis, authority changes, Pyth benchmarks for oracle deviation)
and returns a concrete loss amount + confidence score.

Synthetic / demo trigger signatures (generated by `/api/demo/simulate-exploit`) bypass the verifier
path and flow through a synthetic path gated by `NODE_ENV` and an HMAC-signed alert envelope.

Lock periods per trigger type: exploit=0, oracle_manipulation=1 h, agent_error=6 h, governance_attack=2 h.

## Webhook Authentication

`POST /api/monitoring/webhook` accepts two auth methods:

- **HMAC-SHA256 of body** on `x-helius-hmac-signature` — used by internal callers + tests that
  can compute the signature.
- **Static bearer token** on `Authorization: Bearer <HELIUS_WEBHOOK_SECRET>` — what real Helius
  deliveries actually send (Helius does not HMAC-sign payloads as of 2026-04).

Register the webhook with `pnpm webhook:sync` (idempotent; matches by `webhookURL` and updates
the `accountAddresses` set).

## Reliability & Self-Healing

- **Expiry crank**: `workers/expiry-crank.ts` runs every 60 s + on boot, querying DB-Active-but-
  expired policies and sending `expire_policy` on-chain (cranker = oracle keypair). DB state
  catches up on the next `policy-indexer` tick.
- **Policy indexer**: `workers/policy-indexer.ts` is the only writer for on-chain-authoritative
  fields (`pdaAddress`, `holder`, `agent`, `coverage`, `premium`, `riskTier`, `startTime`,
  `expiryTime`, `state`, `triggerType`, `triggerTxSignature`, `claimSubmittedAt`, `payoutAmount`).
  Its `onConflictDoUpdate` refreshes all of these — so after a program redeploy, stale PDAs
  correct themselves on the next tick.
- **Diagnostics**:
  - `GET /api/monitoring/metrics` — cumulative counters (matched / skipped-uninsured /
    skipped-inactive / anomalies) + `policyLag.stuckCount` / `maxLagSec`.
  - `GET /api/policies/:id/why-active` — side-by-side DB vs on-chain view of a single policy
    with a structured `diagnosis` array and a typed `onChainReason` (`not-found` / `owner-mismatch` /
    `decode-error` / `rpc-error`).
- **UI defensiveness**: the dashboard renders `Expired (pending)` for any policy whose state is
  `Active` but whose `expiry_time` has passed — the truth of the coverage window wins over a
  temporarily-stale badge.

## Security Model

- **Oracle Authority**: Single designated keypair signs `upsert_attestation`, `oracle_submit_claim`,
  and `verify_and_payout`. `create_policy` reads the tier from the oracle-signed `RiskAttestation`
  PDA, so buyers cannot self-select a cheaper tier.
- **Admin Authority**: Protocol parameter management via `update_config` (paused flag, oracle
  rotation).
- **Signed internal alert bus**: `monitoring:alerts` on Redis carries an HMAC-SHA256 envelope
  (`ALERT_HMAC_SECRET`); the claim-keeper refuses unsigned or mismatched alerts.
- **PDA-based**: All state accounts use deterministic PDAs including the versioned
  `InsurancePolicy` layout for forward-compatible deserialization.
- **Solvency Guards**: Automatic policy restrictions at low solvency levels; claim payouts
  fail loudly with `InsufficientVaultBalance` rather than silently under-paying.
